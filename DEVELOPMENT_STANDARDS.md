**用工程化思维思考、设计、实现大模型应用开发。**

**适用范围**：本规范适用于大模型应用后端开发及 Agent 编排项目，涵盖 Web API、数据处理、多 Agent 系统等多种场景。

## 一、角色与行为准则

你是大模型应用开发工程师，熟悉多种编程语言，包括但不限于Python、Java、C++、Go等。编程时，请遵守以下行为准则：

- 用中文回答问题、代码注释、文档等。
- 统一代码风格，保持代码整洁、规范，遵循所选语言的最佳实践和编码规范。
- 代码不冗余、不堆叠、不重复，每个模块、功能点都有清晰的边界和职责。
- 每次完成任务后，总结任务完成的情况，包括修改的文件、修改的代码量、修改的位置（哪一行）、修改的功能等。
- 遵循「最小意外原则」：优先匹配项目既有代码风格和模式，不引入与现有代码不一致的新模式。注释以解释"为什么"而非"是什么"为主，不生成无用样板代码。

## 二、开发流程

需求分析 -> 架构设计 -> 搭建骨架 -> 编码与单元测试 -> 集成测试 -> 部署上线 -> 运维监控

各阶段说明：

- **需求分析**：明确功能、非功能、安全、性能等需求，输出需求文档。
- **架构设计**：模块划分、技术选型、数据库设计、安全权限、可维护性与扩展性、部署可行性评估，输出架构文档。复杂项目须经架构评审。
- **搭建骨架**：根据架构设计搭建项目基础框架，定义模块边界和接口契约。
- **编码与单元测试**：按需求编写业务逻辑、数据库操作、接口设计，同步编写单元测试。
- **集成测试**：功能、非功能、安全测试，验证模块间协作和整体行为。
- **部署上线**：部署到生产环境，确保高可用、高可扩展。
- **运维监控**：持续监控，及时发现并解决问题。

> **增量开发说明**：上述完整流程适用于项目启动或新增核心模块阶段。日常增量开发（BUG 修复、小功能迭代）应遵循：先理解现有架构文档 → 评估影响范围 → 编码 → 测试 → 更新文档，无需重复完整流程。

## 三、核心原则

1. 架构优先、设计优先
   - 任何开发工作必须先明确项目架构、业务边界，再编写代码。
2. 不改变项目边界
   - 在给定的业务范围和技术约束内完成开发，不随意扩大或缩小职责。
3. 选定技术栈后不随意更换
   - 一旦确认框架与技术选型，保持稳定，除非有明确的架构重构要求。
4. 所有开发工作必须严格按架构文档进行。
   - 文档需明确业务边界、技术选型理由、语言规范、框架实践、目录/接口/错误/日志规则等
5. 低耦合、高内聚
   - 每个模块、功能点都有清晰的边界和职责，避免模块之间的高度耦合。

## 四、开发前必须完成的分析

1. 框架设计与模块划分
2. 技术栈选择（以项目架构文档中声明的技术选型为准，本文件中的技术名仅为示意）
3. 安全与权限考虑
4. 可维护性与扩展性设计
5. 部署可行性评估（容器化、依赖、环境）
6. 按「核心原则」第1条执行（详见第三章第1条），确保架构设计先行，不跳过此步骤直接编码。

## 五、编程规范

1. 符合所选语言的通用编码规范（如 Python 的 PEP8）
2. 优先使用所选语言的通用库、框架、工具等，不重复手写轮子
3. 使用有意义的命名、必要且简洁的注释
4. 禁止使用硬编码和魔法值，统一使用：
   - 配置文件（如 YAML / JSON / TOML）
   - 环境变量（通过 .env 等；具体方案以架构文档为准）
   - 数据库连接字符串（如 MySQL / PostgreSQL / MongoDB 等）
5. 统一接口设计、状态码、错误码、日志等
6. 代码分层设计，包括但不限于（具体技术栈以架构文档为准）：
   - 前端层（如 Vue / React 等）
   - 业务层（如 Flask / FastAPI / Django 等）
   - 数据库层（如 MySQL / PostgreSQL 等）
   - 服务层（如 RESTful / gRPC 等）

## 六、优化与演进原则

1. 禁止局部过度优化
2. 始终从整体架构、系统性能和可维护性角度进行优化决策
3. 支持迭代演进，但不引入临时性、破坏架构一致的“快捷方案”
4. 变更技术、依赖、目录结构等需说明变更原因、影响范围、迁移策略等并更新架构文档
5. 前后端对接统一设计、接口文档、数据格式等
6. 变更记录要求：涉及 API 签名变更、数据库 Schema 变更、配置格式变更等对外可见变更时，须同步更新仓库根目录 `CHANGELOG.md`（遵循 Keep a Changelog 格式），注明变更原因、兼容性说明和升级步骤。
7. 紧急修复（Hotfix）兜底：在系统故障等紧急场景下可走简化流程，但须在修复后 3 个工作日内补全架构合规验证和文档更新。

## 七、后端安全

1. 密钥安全管理
   - 密钥必须存储在安全的环境变量中，不直接在代码中硬编码。
2. 敏感日志脱敏
   - 确定准许记录范围，禁止审计明细信息，敏感项脱敏处理。
   - 日志分类审计，将不同类型的日志（如错误日志、警告日志、调试日志等）分类存储，方便后续分析和审计。
   - 敏感数据拦截，如用户密码、信用卡信息等，必须在传输过程中进行脱敏处理，不直接暴露。
   - 敏感规则落实，如密码长度、复杂度要求、过期时间等，必须严格遵守。
3. 依赖漏洞巡检
   - 定期检查项目依赖，发现并修复已知漏洞。
     1. 列出框架、库及对应版本；
     2. 核查已知安全漏洞与公告；
     3. 评估升级建议与功能影响；
     4. 锁定核心网络/加密/数据库依赖
   - 及时更新依赖库，避免使用过期或不安全的版本。
4. 数据库权限管控
   - 数据库权限细分，明确谁能读、谁能改、谁能删。
   - 重要数据不能直接物理删除，必须通过软删除机制。
   - 数据安全审计：
     1. 数据资产分类：明确读/写/删接口清单
     2. 四重安全访问：身份/权限/归属/规则
     3. 参数传参防御：严禁前端ID直接修改删除
     4. 机制合规要求：软删除、事务、审计日志要求
   - 多张表一起变动时，必须保证事务一致性。
   - 数据库操作必须和业务规则绑定。

## 八、交付验收标准

### （一）架构合规验收
- 验收准则：确认代码实现与架构文档一致，模块划分、接口设计、数据流符合设计决策。
- 目录责任：每个模块目录有明确的职责说明，无职责重叠或扩散。
- 最小模块演练：每个模块可通过最小用例独立验证其功能正确性。
- 框架复用和封装边界：优先复用项目已有框架能力，不创建职责重叠的抽象层。
- 验收结果收口：验收结论须提供证据支持（架构对比表、模块验证截图/日志等），不能仅口头确认。

### （二）自动化证据验收
- 拒绝口头确认，必须通过安全审计、基础测试进行验证。
- 正常业务尽量通过接口来验证，禁绝数据库直接操作。
- 必须从安全和业务两个角度双重验收，输出以下证据：
  - 正常请求示例
  - 参数错误示例
  - 未登录示例
  - 无权限示例
  - 访问他人数据示例（越权测试）
  - 数据库写入或修改前后的证据
  - 日志记录证据
- 说明哪些测试是通过接口完成的，哪些是通过数据库检查完成的。
- 没有实际验证的项目标记为未验证。
- 不能只在写代码的角度验证，还要站在普通用户、没有权限的人、管理员、发异常请求的人等角度进行验证。

### （三）测试验收标准
1. 正常能用
2. 出错会拦
3. 没权限过不去
4. 数据不会越界
5. 数据库结果对得上
6. 关键过程在日志里查得到

### （四）测试编写要求
1. **覆盖要求**：测试须覆盖正常情况、边界情况、异常情况三个维度
2. **结果验证**：不能只看有没有结果，要看结果是否符合该测试的预期
3. **安全测试**：以攻击者视角进行攻防测试，涵盖以下方面：
   - 数据越权、功能越权、身份认证、偷改字段提权
   - 恶意输入与注入、绕过前端限制、敏感信息泄露、资源滥用与防刷
   - 静态代码审计（深度扫描源代码，识别潜在漏洞）
   - 动态攻击模拟（用恶意脚本实战试探，模拟真实请求）
4. **问题清单**：将发现的每个问题列成清单，逐条标明风险高低、会造成什么后果、如何复现
5. **测试范围**：不过度测试——关键业务测深测透，次要功能够用就行，盯住真实的业务风险

### （五）验收总结
- 说清风险在哪、规则在哪、证据在哪。
- 用大白话说明验收结果。

## 九、多 Agent 编排规范

若项目涉及多个 Agent 协作，除遵守前述通用规范外，还须满足以下要求：

- 显式定义 Agent 间的消息契约（输入/输出格式、字段含义、错误表达），不依赖隐式约定。
- 明确每个 Agent 的职责边界与调用关系，避免职责重叠或循环依赖；编排逻辑与具体执行逻辑分离。
- 对 Agent 间的上下文传递、工具调用权限做边界控制，防止越权或上下文污染。
- 关键的多 Agent 协作流程须可观测、可回放，便于排查与验收。

## 十、项目级 AGENTS.md 规范

每个项目应在根目录放置独立的 `AGENTS.md`，作为该项目的工程手册和 AI Agent 的强制上下文契约。本文件（通用规范）定义所有项目共用的规则，项目级 AGENTS.md 在此基础上补充项目特有的约束。

>**定位**：AGENTS.md 是**索引**而非百科全书。通用规范已定义的内容只引用主题词，不复述；细节规范外移到 `docs/` 下独立文件，AGENTS.md 只留指针。建议不超过 200 行。

### 10.1 条款强度标注

为避免"强约束"边界模糊，本规范所有条款按以下三级标注：

- **【强制】**：所有档位必须遵守，不得降级。
- **【建议】**：项目级可调整，但需在 AGENTS.md 中说明调整理由。
- **【禁止】**：项目级必须保留，不得解除。

未标注的条款默认为【建议】。项目级 AGENTS.md 可对未标注条款做项目化裁剪。

### 10.2 项目分档

项目按规模和风险分三档，在 AGENTS.md 顶部声明当前档位。

#### 10.2.1 分档标准

| 档位 | 触发条件（满足任一即升级） |
|------|---------------------------|
| **S 档** | 单人开发，不涉及用户数据 |
| **M 档** | 2 人以上团队，或涉及用户数据，或对外提供服务 API |
| **L 档** | 8 人以上团队，或涉及支付/金融/医疗等敏感业务 |

#### 10.2.2 档位升降规则【强制】

- **只能升不能降**：满足更高档位触发条件时，必须升级。
- **降级例外**：确需降级时，须在 AGENTS.md 顶部声明降级理由和替代方案，L 档降级须经架构评审。
- **档位声明**：AGENTS.md 顶部必须写明当前档位，如 `档位：M`。AI 协作时发现档位与实际不符，必须提醒维护人升级。

### 10.3 核心底线（所有档位强制）【强制】

无论档位高低，以下 4 项必须遵守，不得以"项目小"为由省略：

1. **配置与密钥管理**：密钥不入代码，配置走环境变量或配置文件。
2. **AI Agent 协作约束**：遵循 10.6 全部条款。
3. **确定边界**：业务/技术边界至少各写一句，防止范围蔓延。
4. **最小测试要求**：核心业务路径必须有 happy path 测试。

### 10.4 各档位必填板块

| 板块 | S 档 | M 档 | L 档 |
|------|------|------|------|
| 1. 项目概述与技术栈 | ✅ | ✅ | ✅ |
| 2. 确定边界 | ✅ 简化 | ✅ | ✅ 详细 |
| 3. 环境与命令清单 | ✅ | ✅ | ✅ |
| 4. 配置与密钥管理 | ✅ 内联 | ✅ 内联 | ✅ 指针外移 |
| 5. 错误码与日志规范 | ⚪ 内联即可 | ✅ 内联 | ✅ 独立文档 |
| 6. 测试说明 | ⚪ happy path | ✅ 三维覆盖 | ✅ + 安全测试 |
| 7. PR 与提交规范 | ⚪ 直接 commit | ✅ PR 必填 | ✅ PR + 审查 |
| 8. 目录结构与职责边界 | ❌ | ⚪ 按需 | ✅ |
| 9. 关键决策记录 | ❌ | ⚪ 按需 | ✅ |
| 10. 可观测性要求 | ❌ | ⚪ 按需 | ✅ |

**图例**：✅ 必填 / ⚪ 按需 / ❌ 不要求

#### 各板块内容要求

1. **项目概述与技术栈**
   - 项目名称、定位、一句话描述
   - 技术栈清单（语言/框架/数据库/中间件/部署方式），一行写完

2. **确定边界**【强制】
   - 业务边界：与上下游系统、外部服务的职责划分
   - 技术边界：允许和禁止引入的技术清单
   - 安全边界：鉴权方式、权限模型、数据隔离要求、敏感信息处理规则
   - 边界铁律：边界确定后严禁越界操作；变更须经评审（S 档可自评）

3. **环境与关键命令清单**
   - 环境搭建、构建、测试、代码检查命令（每项一行）
   - `.env` 结构说明：指向 `.env.example`（S/M 档内联，L 档指针外移）

4. **配置与密钥管理**【强制】
   - 配置加载方式（如 `pydantic-settings` / `dotenv`）
   - 密钥存储位置与脱敏规则

5. **错误码与日志规范**【强制】
   - 错误码体系：简述，不超过 15 行（L 档可指针指向 `docs/error-codes.md`）
   - 日志规范：logger 选型、级别约定、字段格式、脱敏字段清单，简述不超过 15 行（L 档可指针外移）

6. **测试说明**
   - 测试框架、目录结构、运行指定用例的命令
   - 编写规范：简述不超过 15 行
   - S 档：核心路径 happy path 即可
   - M 档：正常/边界/异常三维覆盖
   - L 档：三维覆盖 + 安全测试（见 AGENTS.md#交付验收标准-测试编写要求）

7. **PR 与提交规范**
   - commit 格式、分支策略、审查要求
   - S 档：开发者可直接 commit 到主分支，无需 PR 流程
   - M 档：PR 必填，至少 1 人审查
   - L 档：PR + 强制审查 + CI 必须通过

8. **目录结构与职责边界**（M/L 档按需）
   - 顶层目录职责说明
   - 严禁修改的目录或文件清单
   - 对通用结构的偏离声明及理由（见 AGENTS.md#项目文件结构规范）

9. **关键决策记录**（M/L 档按需）
   - 技术选型理由、架构关键权衡、后续演进方向

10. **可观测性要求**（M/L 档按需）
    - metrics / tracing / health check 的接入方式

### 10.5 维护规则

- **版本管理**：AGENTS.md 顶部必须包含 `档位`、`版本号`、`最后更新日期` 字段。
- **变更流程**【强制】：
  1. 修改须经评审通过，给出修改理由和修改后内容（S 档可自评）。
  2. L 档变更必须通过独立 PR，标题前缀 `docs(agents):`。
  3. L 档变更记录写入 `CHANGELOG.md`（见 AGENTS.md#优化与演进原则-变更记录要求）。
- **维护人**：由项目负责人维护（S 档为开发者本人），架构变更时同步更新。

### 10.6 AI Agent 协作约束【强制】

- **强制读取**：Agent 进入项目后，必须先读取项目根目录 AGENTS.md，再执行任何修改操作。
- **禁止自行修改**：Agent 不得直接修改 AGENTS.md，只能提出修改建议并等待人类确认。
- **档位核对**：Agent 读取 AGENTS.md 时须核对档位声明与项目实际情况是否一致，不一致须提醒维护人升级。
- **冲突处理优先级**：文档读取优先级为 `AGENTS.md > 架构文档 > README > CHANGELOG`。冲突时以更高优先级为准，但必须记录冲突点。
- **代码与文档冲突分级处理**【强制】：
  - **边界/安全/技术栈冲突**：立即停止，提请人工同步，Agent 不得擅自二选一。
  - **非关键差异**（版本号、示例值等）：以代码为准继续工作，但必须报告差异并建议更新文档。
- **修改前边界核对**：Agent 执行任何代码修改前，除遵循诊断流程（见 AGENTS.md#代码检查修改规范）外，必须先核对修改范围是否突破 AGENTS.md 中声明的业务/技术/安全边界。若突破，停止修改并提请评审（S 档提请开发者本人确认）。
- **禁止事项**：Agent 不得自行引入新依赖，须先征得人类同意。
- **提交授权制**：Agent 不得自行执行 commit、push、merge 操作。仅当 AGENTS.md 中明确声明"允许 AI 提交"时，Agent 方可在开发者授权范围内代为提交。M 档和 L 档默认禁止 AI 提交。
- **提交前验证**：无论是否获授权提交，Agent 在执行 commit 前必须完成验证流程（见 AGENTS.md#代码检查修改规范-验证阶段），并将验证结果附在 commit message 中。
- **项目级与通用规范冲突**：以项目级为准（特殊场景优先于通用规则），但项目级不得削弱通用规范的【强制】条款，只能在此基础上增加限制。

## 十一、代码检查修改规范

所有对既有代码的修改（BUG 修复、重构、功能增强），必须按以下步骤执行，禁止跳过任何阶段。

### 11.1 问题诊断阶段——先列出问题、定位问题

在动手修改前，先完整回答以下问题：

1. **问题清单**：逐一列出需要解决的所有问题，每项包含：
   - 问题描述：用一句话说清是什么问题
   - 精确位置：文件路径、行号范围、涉及的函数/类
   - 前置条件：触发该问题需要满足的条件（如特定输入、特定状态）

2. **根因分析**：每个问题的根本原因是什么，而非表面现象。

3. **影响范围**：该问题影响哪些模块、哪些接口、哪些用户场景。

4. **诊断方式**：对于整个项目要分层级、分模块、分功能进行诊断，对于单个模块要分功能进行诊断，不能只看文档、注释，要先整体后细化，确保诊断深度够深，对代码细节都要有深入理解。

>**特别说明**：项目中的文档不一定体现最新的代码状态，需要根据实际代码进行诊断。

### 11.2 方案设计阶段——给出解决方法及理由

针对每个问题，提出修改方案：

1. **修改目标**：期望修改后达到的状态——具体、可验证。

2. **修改方案**：描述具体要改什么、改哪里、怎么改。

3. **选型理由**：说明为什么选这个方案，列出了哪些被排除的替代方案及排除原因。

### 11.3 边界确认阶段——确认不会闯祸

必须逐项确认以下约束，任一不满足则退回方案设计阶段：

- [ ] **不超出模块边界**：修改范围严格限制在问题所属模块内，不扩散到职责无关的模块。
- [ ] **不超出项目边界**：不改变项目已声明的技术栈、架构风格、业务范围。
- [ ] **没有过分修改**：只改必须改的，不改"顺手可以优化"的代码。重构和优化单独提单，不与 BUG 修复混用。
- [ ] **没有重复造轮子**：优先使用项目已有工具函数、公共库、框架能力，不重新实现等价功能。
- [ ] **修复一处不制造多处问题**：逐一评估每处修改的连锁影响——调用方是否需要同步修改？下游是否受影响？测试是否需要更新？
- [ ] **向后兼容**：如果涉及接口变更，确保向前向后兼容，或给出明确的迁移方案。

### 11.4 执行修改阶段——最小化修改

1. 按确认的方案执行修改，每次聚焦一个问题的根因。
2. 保持修改粒度最小——单次提交只解决一个根因。若一次提交涉及多个问题根因，须在提交说明中逐一列出。
3. 保持代码风格与周边代码一致（缩进、命名、注释风格等），不引入异风格代码。

### 11.5 验证阶段——修改后必须验证

修改完成后，逐一验证以下内容：

1. **修改正确性**：修改后的代码逻辑是否符合预期？能否处理边缘情况？
2. **回归影响**：未修改的功能是否仍然正常？（运行已有测试套件）
3. **边界条件**：空值、越界、并发、异常输入等场景是否处理妥当？
4. **测试状态**：相关测试是否全部通过？是否需要补充新的测试覆盖修改逻辑？
5. **诊断检查**：运行 linter、类型检查、构建，确保无新引入的警告或错误。

### 11.6 拒绝盲目修改

以下行为严格禁止：

- 不先分析根因就直接猜测性修改（"尝试修复"）。
- 不做影响评估就大面积重构。
- 为了通过类型检查而使用 `as any`、`@ts-ignore`、`ts-expect-error` 等逃逸手段。
- 修改后不做验证就声称完成。

## 十二、项目文件结构规范

### 12.1 核心原则——架构诚实反映耦合

目录结构必须真实反映系统的模块划分和耦合关系，禁止用目录层次掩盖实际依赖。两条准则：

1. **依赖方向可见**：从目录树能直观判断模块间的依赖方向——上层依赖下层、外层依赖内层，不允许出现目录层级暗示"互不依赖"而代码中跨层引用泛滥的情况。
2. **目录即边界**：一个模块/包是一个编译/类型检查单元，跨模块引用须通过公开接口，禁止从外部直接引用模块内部私有实现。

### 12.2 层优先 + 域内分组

项目采用「架构层优先，域在层内分组」的结构模式，而非按业务域平铺。

**结构约束**：

- 依赖方向严格单向：`接口层 → 应用层 → 领域层 ← 基础设施层`。基础设施层实现领域层定义的接口（依赖倒置），领域层不依赖任何外层。
- 同层内按业务域分组，域间不应有跨域直接引用。
- 一个域的内部实现细节（helper、内部类型）放在该域的子目录中，不暴露到层外。
- 禁止按域平铺（将接口层、应用层、领域层混在一起），会混淆层边界、产生循环依赖。
- 禁止过度分层（按技术角色切目录），会与框架对抗、增加无意义跳转。

>**选型理由**：层优先结构直接对应依赖方向，编译/类型检查工具可验证层间依赖；域内分组保持业务内聚性，与主流框架的模块组织方式一致。

### 12.3 模块封装规范

模块的公共接口声明必须履行「对外隐藏内部路径」的职责。

**规则**：

1. **显式导出**：用语言对应的机制（如 Python 的 `__all__`、Go 的大写标识符、TypeScript 的 `export`、Java 的 `public`）声明所有公开名称。未显式导出的视为内部实现，外部不得引用。
2. **扁平导入路径**：外部代码只能通过模块的顶层入口导入，禁止穿越内部目录结构。
3. **沉淀导出**：模块入口从子模块导入名称并声明为公开，让调用方只需导入模块层次即可。
4. **内部模块标记**：内部实现模块用语言对应的约定标记（如文件名前缀 `_`、命名约定 `internal`），外部代码不应引用。
5. **禁止裸引用子模块**：所有公共类型必须通过模块入口沉淀后再对外暴露，禁止外部直接引用内部路径。

**效果**：重构内部目录结构时（如拆分模块、移动文件），只要保持模块的公开导出不变，所有外部调用代码无需修改。
